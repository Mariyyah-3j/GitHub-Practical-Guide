# 🟢 المرحلة الأولى: التحكم في النظام (Command Line)

تبدأ الرحلة من هنا…  
قبل أن نتعلم Git و GitHub، كان علينا أولًا أن نفهم البيئة التي نعمل فيها.

---

## 💻 ما هو Command Line؟

الـ **Command Line** هو واجهة نصية تسمح لنا بالتعامل مع نظام التشغيل باستخدام أوامر مكتوبة بدلًا من الواجهة الرسومية (GUI).

يستخدمه المطورون لأنه:

- أسرع في تنفيذ المهام
- يعطي تحكمًا أكبر في الملفات
- ضروري عند العمل مع أدوات مثل Git
- يستخدم في الخوادم والأنظمة الاحترافية

---

## 📁 الفرق بين File و Folder

- **File (ملف):** يحتوي على بيانات، مثل ملف نصي أو صورة أو كود.
- **Folder (مجلد):** يحتوي على ملفات أو مجلدات أخرى لتنظيمها.

تنظيم الملفات بطريقة صحيحة يساعد على:

- سهولة الوصول إليها  
- منع الفوضى  
- إدارة المشاريع بشكل احترافي  

---

## 📂 التنقل داخل النظام

من أهم مهارات هذه المرحلة إتقان التنقل بين المجلدات.

### أهم الأوامر الأساسية:

- `pwd` – يعرض المسار الحالي
- `ls` – يعرض محتويات المجلد
- `cd <folder>` – الانتقال إلى مجلد معين
- `cd ..` – الرجوع خطوة للخلف

📦 إنشاء الملفات والمجلدات
mkdir project-name   # إنشاء مجلد
touch file.txt       # إنشاء ملف
🔄 إدارة الملفات
mv file.txt folder/   # نقل
cp file.txt copy.txt  # نسخ
rm file.txt           # حذف
🧭 فهم المسارات

Absolute Path → مسار كامل يبدأ من جذر النظام

Relative Path → مسار بالنسبة لموقعك الحالي

# 🔵 المرحلة الثانية: فهم Version Control

Version Control هو الأداة الأساسية التي يستخدمها المطورون لتتبع تغييرات الكود عبر الزمن.

### لماذا نحتاجه؟

- حفظ تاريخ جميع التعديلات على الكود
- الرجوع إلى نسخة سابقة عند حدوث خطأ
- معرفة من قام بأي تعديل داخل الفريق
- العمل على نفس المشروع بدون تعارض

> 📌 بدون Version Control قد يتم فقدان العمل أو استبدال ملفات مهمة بالخطأ.

## ما هو Git؟

Git هو نظام لإدارة وتتبع التغييرات في ملفات المشروع. يعمل محليًا على الجهاز ويتيح:

- حفظ تاريخ التعديلات
- الرجوع لنسخ سابقة عند الحاجة
- إنشاء فروع (branches) للعمل على ميزات منفصلة

**مزاياه:**

- سريع
- لا يحتاج اتصال إنترنت
- يعتمد عليه معظم فرق التطوير

## ما هو GitHub؟

GitHub هو منصة على الإنترنت تُستخدم لتخزين مشاريع Git ومشاركتها، وتسهيل التعاون بين المطورين.

- منصة لاستضافة المشاريع
- تعمل عبر الإنترنت
- تتيح إدارة القضايا (issues) والطلبات (pull requests)
- تدعم فرق العمل والتنظيمات

## لماذا يعتمد المطورون على Git؟

يعتمد المطورون عليه لأنه يوفر:

- إنشاء فروع (Branches) لتجربة ميزات جديدة
- العمل الجماعي بدون التأثير على النسخة الرئيسية
- دمج التعديلات بسهولة باستخدام Merge
- تتبع كامل لتاريخ المشروع
- إمكانية العمل offline ثم رفع التعديلات لاحقًا

## كيف يمنع Git ضياع العمل؟

يقوم Git بحفظ نسخة من المشروع في كل مرة نعمل فيها `commit`.

لذلك إذا حدث خطأ أو تم حذف ملف، يمكن الرجوع إلى نسخة سابقة بسهولة واستعادة العمل.

# 🟣 المرحلة الثالثة: أول مشروع باستخدام Git (Local)

🎯 **الهدف:** إنشاء "آلة زمن" للمشروع؛ كل `commit` هو لقطة محفوظة يمكن الرجوع إليها لاحقًا.

## 1. تفعيل Git داخل المشروع
يُطبَّق مرة واحدة لكل مشروع.

```bash
git init
git status
```
> يتم إنشاء مجلد مخفي `.git` الذي يحتفظ بتاريخ المشروع بالكامل.

## 2. إضافة الملفات (Staging)

- لإضافة ملف محدد:
  ```bash
  git add index.html
  ```
- لإضافة جميع الملفات:
  ```bash
  git add .
  ```

المناطق في Git:

- **Working Directory:** الملفات الحالية على القرص.
- **Staging Area:** الملفات المُهيأة للحفظ.
- **Repository:** النسخ المحفوظة (commits).

للتحقق من الحالة:
```bash
git status
```

## 3. حفظ التعديلات (Commit)

```bash
git commit -m "Add login page"
```

✅ **قاعدة الرسائل:** فعل + وصف محدد
- `Add hero section`
- `Fix navbar bug`
- `Update README`

❌ **تجنّب:** `update`، `fix`، `done` بدون تفاصيل.

## 4. مراجعة السجل

```bash
git log
git log --oneline   # مختصر
```

## 5. الروتين الذهبي عند أي تعديل

```bash
git status
git diff
git add .
git commit -m "Describe change"
```

## 6. أوامر مساعدة

- إلغاء الملفات من الـstaging:
  ```bash
  git restore --staged .
  ```
- استعادة ملف إلى حالة ما قبل `commit`:
  ```bash
  git restore filename
  ```

## 7. تمرين سريع

1. أنشئ ملفًا جديدًا.
2. `git add .`
3. `git commit -m "Initial commit"`
4. عدِّل سطرًا داخل الملف.
5. `git status` ثم `git diff` ثم `git add` ثم `git commit -m "Update text"`.
6. اعرض السجل: `git log --oneline`.

# 🟡 المرحلة الرابعة: الانطلاق إلى GitHub

هذه المرحلة تشرح كيف تبدأ وجودك على المنصة وتستخدم أهم الميزات.

## 1. إنشاء حساب GitHub
- اذهب إلى https://github.com
- اضغط **Sign up**
- سجل باستخدام البريد الإلكتروني واتبع الخطوات.

> الآن لديك حساب GitHub جاهز!

## 2. إنشاء Repository جديد
Repository هو المكان الذي تحفظ فيه مشروعك.

- اضغط **New repository**
- أدخل اسم المشروع
- اختر **Private** إذا أردت أن يبقى خاصًّا
- اضغط **Create repository**

## 3. ربط المشروع المحلي بالمستودع
في الترمنال داخل مجلد المشروع:
```bash
git init
git add .
git commit -m "first commit"
# ثم اربط عن بعد
git remote add origin https://github.com/USERNAME/REPO.git
```

## 4. رفع المشروع (git push)
```bash
git branch -M main
git push -u origin main
```
🎉 بهذا تكون رفعت مشروعك إلى GitHub.

## 5. بناء صفحة Profile README
هذا الملف يظهر في صفحة حسابك الرئيسية.

- أنشئ مستودعًا بنفس اسم المستخدم (username/username)
- أضف ملف `README.md`
- ضعه معلومات مثل:
  - نبذة عنك
  - مهاراتك
  - مشاريعك
  - طرق التواصل

## 6. تحسين واجهة الحساب
أضف إلى README عناصر مثل:
- 👨‍💻 من أنا
- 🛠️ التقنيات
- 🚀 المشاريع
- 📫 التواصل

> مثال:
```markdown
# Hi 👋 I'm Mohammad
Frontend Developer
Drupal Developer
```

## 7. ستارز وGists
- **Stars ⭐**: علامة إعجاب بالمشاريع لحفظها والرجوع إليها.
- **Gists 📌**: مكان لحفظ الأكواد القصيرة، الملاحظات، والـsnippets.

يمكنك زيارة https://gist.github.com للتجربة.

## 8. مفاهيم إضافية
- **Projects 📋**: أداة لإدارة المهام داخل GitHub تشبه Trello.
- **Organizations 👥**: فرق أو شركات تنظم مستودعات متعددة.
- **Codespaces 💻**: بيئة تطوير تعمل عبر المتصفح بدون تثبيت.

🎯 الهدف من هذه المرحلة: بناء حضور مهني واحترافي على GitHub.

# 🔴 المرحلة الخامسة: العمل الجماعي والاحتراف

تتناول هذه المرحلة مفاهيم التعاون ومهارات إدارة الفروع والمساهمة بالمشاريع.

## 1. Branching

**ما هو الـ Branching؟**  
هو نسخة منفصلة من المشروع تتيح لك العمل على ميزة أو تعديل دون التأثير على الكود الأساسي.

**لماذا نستخدمه؟**
- يتيح العمل على ميزة جديدة دون كسر المشروع
- يسمح بالتعاون الجماعي بدون تضارب مباشر

## 2. Pull Request

**ما هو Pull Request؟**  
هو طلب دمج التعديلات من فرع (branch) إلى الفرع الرئيسي (`main`).

> 💡 الفكرة الأساسية: منع دمج أي كود مباشرة في الفرع الرئيسي دون مراجعة.

## 3. المراجعة قبل الدمج (Code Review)

**أهمية المراجعة:**
- تحسين جودة الكود
- توحيد أسلوب الكتابة

## 4. استخدام `git clone`

عملية نسخ مشروع موجود على GitHub إلى جهازك المحلي.

**لماذا؟**
- للانضمام إلى مشروع موجود
- للعمل على مشروع مفتوح المصدر
- لتحميل نسخة للعمل محليًا

## 5. الفرق بين Collaborator و Fork

- **Collaborator**: شخص يُمنح صلاحيات مباشرة داخل المستودع. يمكنه تنفيذ تعديلات مباشرة دون الحاجة إلى نسخة منفصلة.
- **Fork**: إنشاء نسخة مستقلة من المشروع في الحساب الشخصي. بعد الانتهاء من التعديلات، يُرسل Pull Request لاقتراح دمج التغييرات.

## 6. إرسال Pull Request لمشاريع خارجية

خطوة متقدمة 💪 وتعتبر بداية دخولك لعالم الـ Open Source.

---

### التطبيق العملي خطوة بخطوة

#### إنشاء branch والعمل عليه

1. التأكد من أنك داخل المشروع: `git status`
2. إنشاء فرع جديد: `git checkout -b feature/login`
3. تنفيذ التعديلات على الملفات
4. حفظ التعديلات:
   ```bash
   git add .
   git commit -m "Add login feature"
   ```
5. رفع الفرع إلى GitHub:
   ```bash
   git push origin feature/login
   ```

📌 الآن صار عندك فرع مستقل يحتوي على التعديلات

#### إنشاء Pull Request عمليًا

1. الدخول إلى المستودع على GitHub.
2. ستظهر رسالة تقترح إنشاء Pull Request.
3. الضغط على **Compare & pull request**.
4. كتابة عنوان ووصف واضحين.
5. الضغط على **Create pull request**.

📌 الآن التعديلات جاهزة للمراجعة.

#### استخدام `git clone` عمليًا

1. نسخ رابط المشروع من GitHub.
2. تنفيذ الأمر:
   ```bash
   git clone https://github.com/username/project.git
   ```
3. حفظه في مجلد.

📌 أصبح المشروع جاهزًا للعمل محليًا.

#### العمل كـ Collaborator

1. عمل clone للمشروع.
2. إنشاء branch.
3. تنفيذ التعديلات.
4. عمل push للفرع.
5. إنشاء Pull Request داخلي.

> لا حاجة إلى fork في هذه الحالة.

#### تنفيذ Fork عمليًا

1. الدخول إلى المشروع المطلوب.
2. الضغط على زر **Fork**.
3. الانتقال إلى النسخة التي أُنشئت في الحساب الشخصي.
4. عمل clone للمشروع.
5. إنشاء branch جديد.
6. تنفيذ التعديل وحفظ التعديلات.
7. رفع التعديلات.
8. الدخول إلى GitHub وإنشاء Pull Request للمستودع الأصلي.
