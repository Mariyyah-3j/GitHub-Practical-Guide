# 🔵 المرحلة الثانية: فهم Version Control

ما هو Version Control؟

هو نظام يتابع كل تعديل على ملفات المشروع.
كل تغيير يُسجل كنسخة للمستخدمين القدرة للرجوع لها في أي وقت.


ما هو Git؟

	•	يحفظ تاريخ كل التعديلات
	•	يسمح للمستخدمين العمل على أكثر من ميزة بدون مداخلته مع المشروع الأساسي (Branches)

المطورون يعتمدون عليه لأنه:
	•	سريع
	•	آمن
	•	يعطي تحكم كامل بتاريخ المشروع


ما هو GitHub؟

GitHub هو منصة على الإنترنت تخزن مشاريع Git.

	•	Git = النظام اللي يدير التعديلات
	•	GitHub = المكان اللذي يُخزن فيه المشروع ويشارك مع الآخرين


كيف يمنع Git ضياع العمل؟
	1.	كل تعديل يُسجل كـ commit
	2.	القدرة على الرجوع لأي نقطة سابقة
	3.	فصل ميزات معينة عن المشروع الرئيسي

# 🟣 المرحلة الثالثة: أول مشروع باستخدام Git

في هذه المرحلة يبدأ التطبيق:
- إنشاء مشروع جديد باستخدام:
  - `git init`
- إضافة الملفات:
  - `git add`
- حفظ التعديلات:
  - `git commit`
- مراجعة السجل:
  - `git log`
- فهم Workflow الصحيح عند إضافة تعديلات جديدة
- حل مشكلة AutoSave

🎯 الهدف من هذه المرحلة:
إدارة نسخ المشروع باحترافية محليًا.

.
. 🚀 المرحلة الخامسة :العمل الجماعي باستخدام Git & GitHub

1️- ما هو الـ Branching؟ 
هو نسخة منفصلة من المشروع تتيح لك العمل على ميزة أو تعديل دون التأثير على الكود الأساسي

  🎯 لماذا نستخدمه؟ 
  • بتيح العمل على ميزة جديدة بدون كسر المشروع 
  •يتيح العمل الجماعي بدون تضارب مباشر 
  •ينظم التطوير داخل الفريق

2- ما هو Pull Request ؟ 
الـPull Request هو طلب دمج التعديلات من Branch إلى الفرع الرئيسي ( main ).

🎯 لماذا نستخدمه؟ 
• لمراجعة الكود قبل الدمج 
• لمناقشة التعديلات 
• لاكتشاف الأخطاء مبكرًا 
• لضمان جودة المشروع 

💡 الفكرة الأساسية
: منع دمج أي كود مباشرة في الـmain بدون مراجعة.

3️- المراجعة قبل الدمج (Code Review)

 🎯 ماهي اهميته؟ 
 • تحسين جودة الكود 
 • مشاركة المعرفة بين الفريق 
 • منع الأخطاء 
 • توحيد أسلوب الكتابة

4️- استخدام git clone
 هو نسخ مشروع موجود على GitHub إلى جهازك . 

 🎯لماذا نستخدمه؟ 
 • للانضمام لمشروع موجود 
 • للعمل على مشروع Open Source 
 • لتحميل نسخة للعمل محليًا 

 5️- الفرق بين Collaborator و Fork
الـ Collaborator هو شخص يُمنح صلاحيات مباشرة داخل المستودع. 

عند إضافته كمشارك رسمي، يمكنه: 
• إنشاء فروع جديدة داخل المشروع 
• تنفيذ تعديلات مباشرة 
• رفع التعديلات (Push) حسب مستوى الصلاحيات 
• المشاركة في إدارة المشروع
يستخدم هذا الأسلوب غالبًا في المشاريع الجامعية الجماعية وفي هذه الحالة، يكون العمل مباشرًا داخل المستودع الأصلي دون الحاجة إلى إنشاء نسخة منفصلة.

الـ Fork 
هو إنشاء نسخة مستقلة من المشروع في حساب شخصي. عند استخدام Fork: 

• لا توجد صلاحيات مباشرة على المشروع الأصلي 
• يتم العمل على نسخة خاصة 
• بعد الانتهاء من التعديلات، يُرسل Pull Request لاقتراح دمج التغييرات




6- إرسال Pull Request لمشاريع خارجية هذه 

خطوة متقدمة 💪 وتعتبر بداية دخولك لعالم الـ Open Source

 الخطوات: 
 • الالتزام بأسلوب المشروع 
 • كتابة رسالة Commit واضحة 
 • كتابة التعديل مع وصفه داخل الـPR

7- ما هو المشروع مفتوح المصدر (Open Source)؟ 
الـ Open Source يعني أن الكود متاح للجميع للمشاهدة والمساهمة.


بعد فهم المفاهيم النظرية، تأتي الخطوة الأهم: 🚀 التطبيق العملي خطوة بخطوة .


 اولًا إنشاء Branch والعمل عليه عمليًا:

التأكد من أنك داخل المشروع: git status
إنشاء فرع جديد: git checkout -b feature/login
تنفيذ التعديلات على الملفات.
حفظ التعديلات: git add . git commit -m "Add login feature رسالة التعديل مثل "
رفع الفرع إلى GitHub: git push origin feature/login -اسم البرانش او الفرع- 

📌 الآن صار عندك فرع مستقل يحتوي على التعديلات 


ثانيًا إنشاء Pull Request عمليًا:

الدخول إلى المستودع.
ستظهر رسالة تقترح إنشاء Pull Request.
الضغط على Compare & Pull Request.
كتابة عنوان واضح.
كتابة وصف يشرح:
ماذا تم تنفيذه؟
وصف ما تم تنفيذه
الضغط على Create Pull Request

📌 الآن التعديلات جاهزة للمراجعة.

ثالثًا تنفيذ Code Review عمليًا:

عند مراجعة PR: 

• قراءة التعديلات سطرًا بسطر. 
• إضافة تعليق عند الحاجة. 
• اقتراح تحسينات. 
• طلب تعديل (Request Changes) أو الموافقة (Approve). عند استلام ملاحظات:
تنفيذ التعديلات المطلوبة.
عمل commit جديد.
رفع التحديث: git push origin feature/login و سيتم تحديث الـ PR تلقائيًا.

رابعًا استخدام git clone عمليًا الخطوات:

نسخ رابط المشروع من GitHub.
تنفيذ الأمر: git clone https://github.com/username/project.git نسخ رابط المشروع
حفظه في المجلد:

📌 أصبح المشروع جاهزًا للعمل محليًا.

خامسًا العمل كـ Collaborator
 (خطوات عملية)
  إذا تمت إضافتك كمشارك: 1
 - عمل clone للمشروع 2
 . إنشاء Branch. 3.
  تنفيذ التعديلات. 4
  . عمل Push للفرع. 5
  . إنشاء Pull Request داخلي. 

  لا حاجة إلى  Fork في هذه الحالة

سادسًا تنفيذ Fork عمليًا

الدخول إلى المشروع المطلوب.
الضغط على زر Fork.
الانتقال إلى النسخة التي أُنشئت في الحساب الشخصي.
عمل clone للمشروع
إنشاء Branch جديد: git checkout -b fix/typo اسم البرانش
تنفيذ التعديل.
حفظ التعديلات: git add . git commit -m "Fix typo in README" رسالة التعديل
رفع التعديلات: git push origin fix/typo اسم البرانش
الدخول إلى GitHub وإنشاء Pull Request للمستودع الأصلي.
