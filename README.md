# 🟢 المرحلة الأولى: التحكم في النظام (Command Line)

تبدأ الرحلة من هنا…  
قبل أن نتعلم Git و GitHub، كان علينا أولًا أن نفهم البيئة التي نعمل فيها.

---

## 💻 ما هو Command Line؟

الـ **Command Line** هو واجهة نصية تسمح لنا بالتعامل مع نظام التشغيل باستخدام أوامر مكتوبة بدلًا من الواجهة الرسومية (GUI).

يستخدمه المطورون لأنه:

- أسرع في تنفيذ المهام
- يعطي تحكمًا أكبر في الملفات
- ضروري عند العمل مع أدوات مثل Git
- يستخدم في الخوادم والأنظمة الاحترافية

---

## 📁 الفرق بين File و Folder

- **File (ملف):** يحتوي على بيانات، مثل ملف نصي أو صورة أو كود.
- **Folder (مجلد):** يحتوي على ملفات أو مجلدات أخرى لتنظيمها.

تنظيم الملفات بطريقة صحيحة يساعد على:

- سهولة الوصول إليها  
- منع الفوضى  
- إدارة المشاريع بشكل احترافي  

---

## 📂 التنقل داخل النظام

من أهم مهارات هذه المرحلة إتقان التنقل بين المجلدات.

### أهم الأوامر الأساسية:

- `pwd` – يعرض المسار الحالي
- `ls` – يعرض محتويات المجلد
- `cd <folder>` – الانتقال إلى مجلد معين
- `cd ..` – الرجوع خطوة للخلف

# 🔵 المرحلة الثانية: فهم Version Control
---

## ما هو Version Control؟

هو نظام يتابع كل تعديل على ملفات المشروع. كل تغيير يُسجل كنسخة، مما يمنح المستخدمين القدرة على الرجوع لأي نقطة سابقة.

## ما هو Git؟

Git هو نظام إدارة الإصدارات الأساسي الذي يساعد على:

- حفظ تاريخ كل التعديلات
- العمل على ميزات متعددة عبر فروع (branches)

**لماذا يستخدمه المطورون؟**

- سريع
- آمن
- يوفر تحكمًا كاملًا بتاريخ المشروع

## ما هو GitHub؟

GitHub هو منصة على الإنترنت تخزن مشاريع Git.

- **Git**: النظام الذي يدير التعديلات
- **GitHub**: المكان الذي يُخزن فيه المشروع ويشارك مع الآخرين

## كيف يمنع Git ضياع العمل؟

1. كل تعديل يُسجل كـ commit
2. القدرة على الرجوع لأي نقطة سابقة
3. فصل ميزات معينة عن المشروع الرئيسي


---

# 🟣 المرحلة الثالثة: أول مشروع باستخدام Git

في هذه المرحلة يبدأ التطبيق:

- إنشاء مشروع جديد باستخدام `git init`
- إضافة الملفات باستخدام `git add`
- حفظ التعديلات بواسطة `git commit`
- مراجعة السجل عبر `git log`
- فهم Workflow الصحيح عند إضافة تعديلات جديدة
- حل مشكلة AutoSave

🎯 الهدف من هذه المرحلة:
إدارة نسخ المشروع باحترافية محليًا.

---

# 🟡 المرحلة الرابعة: الانطلاق إلى GitHub

في هذه المرحلة يتم:

- إنشاء حساب على GitHub
- إنشاء Repository خاص
- ربط المشروع المحلي بـ GitHub
- رفع المشروع باستخدام `git push`
- فهم Profile README
- إنشاء واجهة احترافية للحساب
- التعرف على Stars و Gist
- فهم Projects و Organizations و Codespaces

🎯 الهدف من هذه المرحلة:
بناء حضور احترافي على GitHub.

# 🔴 المرحلة الخامسة: العمل الجماعي والاحتراف

تتناول هذه المرحلة مفاهيم التعاون ومهارات إدارة الفروع والمساهمة بالمشاريع.

## 1. Branching

**ما هو الـ Branching؟**  
هو نسخة منفصلة من المشروع تتيح لك العمل على ميزة أو تعديل دون التأثير على الكود الأساسي.

**لماذا نستخدمه؟**
- يتيح العمل على ميزة جديدة دون كسر المشروع
- يسمح بالتعاون الجماعي بدون تضارب مباشر

## 2. Pull Request

**ما هو Pull Request؟**  
هو طلب دمج التعديلات من فرع (branch) إلى الفرع الرئيسي (`main`).

> 💡 الفكرة الأساسية: منع دمج أي كود مباشرة في الفرع الرئيسي دون مراجعة.

## 3. المراجعة قبل الدمج (Code Review)

**أهمية المراجعة:**
- تحسين جودة الكود
- توحيد أسلوب الكتابة

## 4. استخدام `git clone`

عملية نسخ مشروع موجود على GitHub إلى جهازك المحلي.

**لماذا؟**
- للانضمام إلى مشروع موجود
- للعمل على مشروع مفتوح المصدر
- لتحميل نسخة للعمل محليًا

## 5. الفرق بين Collaborator و Fork

- **Collaborator**: شخص يُمنح صلاحيات مباشرة داخل المستودع. يمكنه تنفيذ تعديلات مباشرة دون الحاجة إلى نسخة منفصلة.
- **Fork**: إنشاء نسخة مستقلة من المشروع في الحساب الشخصي. بعد الانتهاء من التعديلات، يُرسل Pull Request لاقتراح دمج التغييرات.

## 6. إرسال Pull Request لمشاريع خارجية

خطوة متقدمة 💪 وتعتبر بداية دخولك لعالم الـ Open Source.

---

### التطبيق العملي خطوة بخطوة

#### إنشاء branch والعمل عليه

1. التأكد من أنك داخل المشروع: `git status`
2. إنشاء فرع جديد: `git checkout -b feature/login`
3. تنفيذ التعديلات على الملفات
4. حفظ التعديلات:
   ```bash
   git add .
   git commit -m "Add login feature"
   ```
5. رفع الفرع إلى GitHub:
   ```bash
   git push origin feature/login
   ```

📌 الآن صار عندك فرع مستقل يحتوي على التعديلات

#### إنشاء Pull Request عمليًا

1. الدخول إلى المستودع على GitHub.
2. ستظهر رسالة تقترح إنشاء Pull Request.
3. الضغط على **Compare & pull request**.
4. كتابة عنوان ووصف واضحين.
5. الضغط على **Create pull request**.

📌 الآن التعديلات جاهزة للمراجعة.

#### استخدام `git clone` عمليًا

1. نسخ رابط المشروع من GitHub.
2. تنفيذ الأمر:
   ```bash
   git clone https://github.com/username/project.git
   ```
3. حفظه في مجلد.

📌 أصبح المشروع جاهزًا للعمل محليًا.

#### العمل كـ Collaborator

1. عمل clone للمشروع.
2. إنشاء branch.
3. تنفيذ التعديلات.
4. عمل push للفرع.
5. إنشاء Pull Request داخلي.

> لا حاجة إلى fork في هذه الحالة.

#### تنفيذ Fork عمليًا

1. الدخول إلى المشروع المطلوب.
2. الضغط على زر **Fork**.
3. الانتقال إلى النسخة التي أُنشئت في الحساب الشخصي.
4. عمل clone للمشروع.
5. إنشاء branch جديد.
6. تنفيذ التعديل وحفظ التعديلات.
7. رفع التعديلات.
8. الدخول إلى GitHub وإنشاء Pull Request للمستودع الأصلي.
