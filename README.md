- إنشاء المجلدات والملفات

- الفرق بين Absolute Path و Relative Path

# 🟣 المرحلة الثالثة: أول مشروع باستخدام Git

في هذه المرحلة يبدأ التطبيق:
- إنشاء مشروع جديد باستخدام:
  - `git init`
- إضافة الملفات:
  - `git add`
- حفظ التعديلات:
  - `git commit`
- مراجعة السجل:
  - `git log`
- فهم Workflow الصحيح عند إضافة تعديلات جديدة
- حل مشكلة AutoSave

🎯 الهدف من هذه المرحلة:
إدارة نسخ المشروع باحترافية محليًا.

- إنشاء واجهة احترافية للحساب

- فهم Projects و Organizations و Codespaces

بناء حضور احترافي على GitHub.

.
🚀 المرحلة الخامسة  :العمل الجماعي باستخدام Git & GitHub

1️⃣ ما هو الـ Branching؟ ولماذا هو مهم؟
الـ  Branch هو نسخة منفصلة من المشروع تتيح لك العمل على ميزة أو تعديل دون التأثير على الكود الأساسي (main)
🎯 لماذا نستخدمه؟
•	العمل على ميزة جديدة بدون كسر المشروع
•	إمكانية التجربة بحرية
•	العمل الجماعي بدون تضارب مباشر
•	تنظيم التطوير داخل الفريق
🛠 مثال عملي:
git checkout -b feature/login 2️⃣ 

ما هو Pull Request ؟
الـ  Pull Request هو طلب دمج التعديلات من  Branch إلى الفرع الرئيسي.
🎯 لماذا نستخدمه؟
•	مراجعة الكود قبل الدمج
•	مناقشة التعديلات
•	اكتشاف الأخطاء مبكرًا
•	ضمان جودة المشروع
💡 الفكرة الذهبية:
لا يتم دمج أي كود مباشرة في   main بدون مراجعة.

3️⃣ المراجعة قبل الدمج (Code Review)
🎯 ماهي اهميته؟
•	تحسين جودة الكود
•	مشاركة المعرفة بين الفريق
•	منع الأخطاء
•	توحيد أسلوب الكتابة

4️⃣ استخدام git clone
هو نسخ مشروع موجود على GitHub إلى جهازك.
 🎯لماذا نستخدمه؟
•	الانضمام لمشروع موجود
•	العمل على مشروع Open Source
•	تحميل نسخة للعمل محليًا
•	
5️⃣ الفرق بين   Collaborator و Fork
الـ Collaborator هو شخص يُمنح صلاحيات مباشرة داخل المستودع.
عند إضافته كمشارك رسمي، يمكنه:
•	إنشاء فروع جديدة داخل المشروع
•	تنفيذ تعديلات مباشرة
•	رفع التعديلات (Push) حسب مستوى الصلاحيات
•	المشاركة في إدارة المشروع  
يستخدم هذا الأسلوب غالبًا في المشاريع الجامعية الجماعية وفي هذه الحالة، يكون العمل مباشرًا داخل المستودع الأصلي دون الحاجة إلى إنشاء نسخة منفصلة.
________________________________________
الـ Fork هو إنشاء نسخة مستقلة من المشروع في حساب شخصي.
عند استخدام Fork:
•	لا توجد صلاحيات مباشرة على المشروع الأصلي
•	يتم العمل على نسخة خاصة
•	بعد الانتهاء من التعديلات، يُرسل Pull Request لاقتراح دمج التغييرات

6️⃣ ما هو Fork Workflow؟
الخطوات:
1.	عمل Fork للمشروع
2.	Clone النسخة الخاصة بك
3.	إنشاء Branch
4.	تنفيذ التعديلات
5.	Push إلى مستودعك
6.	إنشاء Pull Request للمشروع الأصلي
7️⃣ إرسال  Pull Request لمشاريع خارجية
هذه خطوة متقدمة 💪
وتعتبر بداية دخولك لعالم الـ Open Source
الخطوات:
•	الالتزام بأسلوب المشروع
•	كتابة رسالة Commit  واضحة
•	شرح التعديل مع وصفه داخل الـ PR
8️⃣ ما هو Open Source؟
الـ Open Source  يعني أن الكود متاح للجميع للمشاهدة والمساهمة.
المساهمة في هذه المشاريع ترفع مستواك التقني و تبني سمعتك كمطوّر

بعد فهم المفاهيم النظرية، تأتي الخطوة الأهم: التطبيق العملي خطوة بخطوة.
1️⃣ إنشاء Branch والعمل عليه عمليًا
✅ الخطوات العملية:
1.	التأكد من أنك داخل المشروع:
git status
2.	إنشاء فرع جديد:
git checkout -b feature/login
3.	تنفيذ التعديلات على الملفات.
4.	حفظ التعديلات:
git add .
git commit -m "Add login feature"
5.	رفع الفرع إلى GitHub:
git push origin feature/login
📌 الآن أصبح لديك فرع مستقل يحتوي على التعديلات.
________________________________________
2️⃣ إنشاء Pull Request عمليًا
✅ الخطوات على GitHub:
1.	الدخول إلى المستودع.
2.	ستظهر رسالة تقترح إنشاء Pull Request.
3.	الضغط على Compare & Pull Request.
4.	كتابة عنوان واضح.
5.	كتابة وصف يشرح:
o	ماذا تم تنفيذه؟
o	لماذا تم التنفيذ؟
o	هل هناك نقاط تحتاج مراجعة؟
6.	الضغط على Create Pull Request.
📌 الآن التعديلات جاهزة للمراجعة.
________________________________________
3️⃣ تنفيذ Code Review عمليًا
عند مراجعة PR:
•	قراءة التعديلات سطرًا بسطر.
•	إضافة تعليق عند الحاجة.
•	اقتراح تحسينات.
•	طلب تعديل (Request Changes) أو الموافقة (Approve).
عند استلام ملاحظات:
1.	تنفيذ التعديلات المطلوبة.
2.	عمل commit جديد.
3.	رفع التحديث:
git push origin feature/login
سيتم تحديث الـ PR تلقائيًا.
________________________________________
4️⃣ استخدام git clone عمليًا
الخطوات:
1.	نسخ رابط المشروع من GitHub.
2.	تنفيذ الأمر:
git clone https://github.com/username/project.git
3.	الدخول إلى المجلد:
cd project
📌 أصبح المشروع جاهزًا للعمل محليًا.
________________________________________
5️⃣ العمل كـ Collaborator )خطوات عملية(
إذا تمت إضافتك كمشارك:
1.	Clone المشروع.
2.	إنشاء Branch.
3.	تنفيذ التعديلات.
4.	Push الفرع.
5.	إنشاء Pull Request داخلي.
لا حاجة إلى Fork في هذه الحالة.
________________________________________
6️⃣ تنفيذ Fork Workflow عمليًا
✅ الخطوات بالتسلسل:
1.	الدخول إلى المشروع المطلوب.
2.	الضغط على زر Fork.
3.	الانتقال إلى النسخة التي أُنشئت في الحساب الشخصي.
4.	تنفيذ:
git clone https://github.com/username/project.git
5.	إنشاء Branch جديد:
git checkout -b fix/typo
6.	تنفيذ التعديل.
7.	حفظ التعديلات:
git add .
git commit -m "Fix typo in README"
8.	رفع التعديلات:
git push origin fix/typo
9.	الدخول إلى GitHub وإنشاء Pull Request للمستودع الأصلي.
________________________________________
7️⃣ إرسال Pull Request لمشروع خارجي باحتراف
قبل الإرسال:
•	قراءة ملف CONTRIBUTING.md
•	التأكد من تنسيق الكود
•	اختبار التعديلات محليًا
عند كتابة PR:
•	عنوان واضح ومختصر
•	شرح المشكلة
•	شرح الحل
•	توضيح التأثير على المشروع
