# 🔵 المرحلة الثانية: فهم Version Control

ما هو Version Control؟

هو نظام يسمح للمطورين بتتبع جميع التغييرات التي تحدث على ملفات المشروع مع مرور الوقت.

يساعد Version Control على:
	•	حفظ تاريخ جميع التعديلات على الكود
	•	الرجوع إلى نسخة سابقة عند حدوث خطأ
	•	معرفة من قام بأي تعديل داخل الفريق
	•	العمل على نفس المشروع بدون تعارض

📌 بدون Version Control قد يتم فقدان العمل أو استبدال ملفات مهمة بالخطأ

ما هو Git?
Git هو نظام لإدارة وتتبع التغييرات في ملفات المشروع، يساعد المطورين على حفظ تاريخ التعديلات والرجوع لنسخ سابقة عند الحاجة.

  • نظام لإدارة الإصدارات
	• يعمل محليًا على الجهاز
  • يتتبع تغييرات الكود
  • لا يحتاج إنترنت

✅ لماذا يعتمد المطورون على Git في إدارة مشاريعهم؟

يعتمد المطورون على Git لأنه يوفر:
	•	إنشاء فروع (Branches) لتجربة ميزات جديدة
	•	العمل الجماعي بدون التأثير على النسخة الرئيسية
	•	دمج التعديلات بسهولة باستخدام Merge
	•	تتبع كامل لتاريخ المشروع
	•	إمكانية العمل Offline ثم رفع التعديلات لاحقًا

✅ كيف يمنع Git ضياع العمل؟

يقوم Git بحفظ نسخة من المشروع في كل مرة نعمل فيها commit.

لذلك إذا حدث خطأ أو تم حذف ملف، يمكن الرجوع إلى نسخة سابقة بسهولة واستعادة العمل.

ما هو GitHub؟
GitHub هو منصة على الإنترنت تُستخدم لتخزين مشاريع Git ومشاركتها، وتسهيل التعاون بين المطورين والعمل الجماعي على نفس المشروع.

	•	منصة لاستضافة المشاريع
  • يعمل عبر الإنترنت
  • يسهل التعاون بين المطورين
  • يحتاج اتصال بالإنترنت

# 🔴 المرحلة الخامسة: العمل الجماعي والاحتراف
.

1️- ما هو الـ Branching؟ 
هو نسخة منفصلة من المشروع تتيح لك العمل على ميزة أو تعديل دون التأثير على الكود الأساسي

  🎯 لماذا نستخدمه؟ 
  • بتيح العمل على ميزة جديدة بدون كسر المشروع 
  •يتيح العمل الجماعي بدون تضارب مباشر 
  
2- ما هو Pull Request ؟ 
الـPull Request هو طلب دمج التعديلات من Branch إلى الفرع الرئيسي ( main ).

💡 الفكرة الأساسية
منع دمج أي كود مباشرة في الـmain بدون مراجعة.

3️- المراجعة قبل الدمج (Code Review)

 🎯 ماهي اهميته؟ 
 • تحسين جودة الكود 
 • توحيد أسلوب الكتابة

4️- استخدام git clone
 هو نسخ مشروع موجود على GitHub إلى جهازك . 

 🎯لماذا نستخدمه؟ 
 • للانضمام لمشروع موجود 
 • للعمل على مشروع Open Source 
 • لتحميل نسخة للعمل محليًا 

 5️- الفرق بين Collaborator و Fork
الـ Collaborator هو شخص يُمنح صلاحيات مباشرة داخل المستودع. 

عند إضافته كمشارك رسمي، يمكنه: 
• تنفيذ تعديلات مباشرة دون الحاجة إلى إنشاء نسخة منفصلة.

الـ Fork 
هو إنشاء نسخة مستقلة من المشروع في حساب شخصي.  
• بعد الانتهاء من التعديلات، يُرسل Pull Request لاقتراح دمج التغييرات


6- إرسال Pull Request لمشاريع خارجية هذه 

خطوة متقدمة 💪 وتعتبر بداية دخولك لعالم الـ Open Source

بعد فهم المفاهيم النظرية، تأتي الخطوة الأهم: 🚀 **التطبيق العملي خطوة بخطوة**.


 اولًا إنشاء Branch والعمل عليه عمليًا:

1- التأكد من أنك داخل المشروع: git status
2- إنشاء فرع جديد: git checkout -b feature/login
3- تنفيذ التعديلات على الملفات.
4- حفظ التعديلات: git add . git commit -m "Add login feature رسالة التعديل مثل "
5- رفع الفرع إلى GitHub: git push origin feature/login -اسم البرانش او الفرع- 

📌 الآن صار عندك فرع مستقل يحتوي على التعديلات 

ثانيًا إنشاء Pull Request عمليًا:
1- الدخول إلى المستودع.
2- ستظهر رسالة تقترح إنشاء Pull Request.
3- الضغط على Compare & Pull Request.
4- كتابة عنوان ووصف واضحين.
5- الضغط على Create Pull Request

📌 الآن التعديلات جاهزة للمراجعة.

ثالثًا استخدام git clone عمليًا الخطوات:

1- نسخ رابط المشروع من GitHub.
2- تنفيذ الأمر: git clone https://github.com/username/project.git 
3- حفظه في مجلد

📌 أصبح المشروع جاهزًا للعمل محليًا.

خامسًا العمل كـ Collaborator
 (خطوات عملية)
 1- عمل clone للمشروع 
 2- إنشاء Branch.
 3- تنفيذ التعديلات. 
 4-  عمل Push للفرع. 
 5- إنشاء Pull Request داخلي. 

  لا حاجة إلى  Fork في هذه الحالة

**سادسًا** تنفيذ Fork عمليًا

1- الدخول إلى المشروع المطلوب.
2- الضغط على زر Fork.
3- الانتقال إلى النسخة التي أُنشئت في الحساب الشخصي.
4- عمل clone للمشروع.
5- إنشاء Branch جديد .
6- تنفيذ التعديل وحفظ التعديلات.
7- رفع التعديلات.
8- الدخول إلى GitHub وإنشاء Pull Request للمستودع الأصلي.
