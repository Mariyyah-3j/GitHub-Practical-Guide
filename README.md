# 🟣 المرحلة الثالثة: أول مشروع باستخدام Git

في هذه المرحلة يبدأ التطبيق:
- إنشاء مشروع جديد باستخدام:
  - `git init`
- إضافة الملفات:
  - `git add`
- حفظ التعديلات:
  - `git commit`
- مراجعة السجل:
  - `git log`
- فهم Workflow الصحيح عند إضافة تعديلات جديدة
- حل مشكلة AutoSave

🎯 الهدف من هذه المرحلة:
إدارة نسخ المشروع باحترافية محليًا.

🎯 الهدف من هذه المرحلة:
تطبيق رحلة المطور الحقيقية في بيئة عمل جماعية.

.
🚀 المرحلة الخامسة  :العمل الجماعي باستخدام Git & GitHub

1️- ما هو الـ Branching؟ ولماذا هو مهم؟
الـ Branch هو نسخة منفصلة من المشروع تتيح لك العمل على ميزة أو تعديل دون التأثير على الكود الأساسي (main)
🎯 لماذا نستخدمه؟
•	بتيح العمل على ميزة جديدة بدون كسر المشروع
•	يمكن من التجربة بحرية
•يتيح	العمل الجماعي بدون تضارب مباشر
•ينظم التطوير داخل الفريق

2- ما هو Pull Request ؟
الـPull Request هو طلب دمج التعديلات من  Branch إلى الفرع الرئيسي ( main ).

🎯 لماذا نستخدمه؟
•	لمراجعة الكود قبل الدمج
•	لمناقشة التعديلات
•	لاكتشاف الأخطاء مبكرًا
•	لضمان جودة المشروع
💡 الفكرة الأساسية:
 منع دمج أي كود مباشرة في الـmain بدون مراجعة.

3️- المراجعة قبل الدمج (Code Review)
🎯 ماهي اهميته؟
•	تحسين جودة الكود
•	مشاركة المعرفة بين الفريق
•	منع الأخطاء
•	توحيد أسلوب الكتابة

4️- استخدام git clone
هو نسخ مشروع موجود على GitHub إلى جهازك
.
 🎯لماذا نستخدمه؟
•	للانضمام لمشروع موجود
•	للعمل على مشروع Open Source
•	لتحميل نسخة للعمل محليًا
•	
5️- الفرق بين Collaborator و Fork
-
-  الـ Collaborator هو شخص يُمنح صلاحيات مباشرة داخل المستودع.
عند إضافته كمشارك رسمي، يمكنه:
•	إنشاء فروع جديدة داخل المشروع
•	تنفيذ تعديلات مباشرة
•	رفع التعديلات (Push) حسب مستوى الصلاحيات
•	المشاركة في إدارة المشروع  
يستخدم هذا الأسلوب غالبًا في المشاريع الجامعية الجماعية وفي هذه الحالة، يكون العمل مباشرًا داخل المستودع الأصلي دون الحاجة إلى إنشاء نسخة منفصلة.
________________________________________
- الـ Fork هو إنشاء نسخة مستقلة من المشروع في حساب شخصي.
عند استخدام Fork:
•	لا توجد صلاحيات مباشرة على المشروع الأصلي
•	يتم العمل على نسخة خاصة
•	بعد الانتهاء من التعديلات، يُرسل Pull Request لاقتراح دمج التغييرات


6-  إرسال  Pull Request لمشاريع خارجية
هذه خطوة متقدمة 💪
وتعتبر بداية دخولك لعالم الـ Open Source
الخطوات:
•	الالتزام بأسلوب المشروع
•	كتابة رسالة Commit  واضحة
•	كتابة التعديل مع وصفه داخل الـPR 

7-  ما هو المشروع مفتوح المصدر (Open Source)؟
الـ Open Source  يعني أن الكود متاح للجميع للمشاهدة والمساهمة.


بعد فهم المفاهيم النظرية، تأتي الخطوة الأهم:
 🚀 التطبيق العملي خطوة بخطوة
 .
اولًا إنشاء Branch والعمل عليه عمليًا:
1.	التأكد من أنك داخل المشروع:
git status
2.	إنشاء فرع جديد:
git checkout -b feature/login
3.	تنفيذ التعديلات على الملفات.
4.	حفظ التعديلات:
git add .
git commit -m "Add login feature  رسالة التعديل مثل "
5.	رفع الفرع إلى GitHub:
git push origin feature/login -اسم البرانش او الفرع-
📌 الآن صار عندك فرع مستقل يحتوي على التعديلات.
________________________________________
ثانيًا إنشاء Pull Request عمليًا: 
1.	الدخول إلى المستودع.
2.	ستظهر رسالة تقترح إنشاء Pull Request.
3.	الضغط على Compare & Pull Request.
4.	كتابة عنوان واضح.
5.	كتابة وصف يشرح:
-  ماذا تم تنفيذه؟
-  وصف ما تم تنفيذه 
7.	الضغط على Create Pull Request
   
📌 الآن التعديلات جاهزة للمراجعة.
________________________________________
ثالثًا تنفيذ Code Review عمليًا: 

عند مراجعة PR:
•	قراءة التعديلات سطرًا بسطر.
•	إضافة تعليق عند الحاجة.
•	اقتراح تحسينات.
•	طلب تعديل (Request Changes) أو الموافقة (Approve).
عند استلام ملاحظات:
1.	تنفيذ التعديلات المطلوبة.
2.	عمل commit جديد.
3.	رفع التحديث:
git push origin feature/login
و سيتم تحديث الـ PR تلقائيًا.
________________________________________
رابعًا استخدام git clone عمليًا
الخطوات:
1.	نسخ رابط المشروع من GitHub.
2.	تنفيذ الأمر:
git clone https://github.com/username/project.git نسخ رابط المشروع
3.	حفظه في المجلد:

📌 أصبح المشروع جاهزًا للعمل محليًا.
________________________________________
خامسًا العمل كـ Collaborator (خطوات عملية)
إذا تمت إضافتك كمشارك:
1- عمل clone للمشروع
2.	إنشاء Branch.
3.	تنفيذ التعديلات.
4.	عمل Push للفرع.
5.	إنشاء Pull Request داخلي.
لا حاجة إلى Fork في هذه الحالة.
________________________________________
سادسًا تنفيذ Fork عمليًا
1.	الدخول إلى المشروع المطلوب.
2.	الضغط على زر Fork.
3.	الانتقال إلى النسخة التي أُنشئت في الحساب الشخصي.
4.  عمل clone للمشروع

5.	إنشاء Branch جديد:
git checkout -b fix/typo اسم البرانش
6.	تنفيذ التعديل.
7.	حفظ التعديلات:
git add .
git commit -m "Fix typo in README" رسالة التعديل 
8.	رفع التعديلات:
git push origin fix/typo اسم البرانش 
9.	الدخول إلى GitHub وإنشاء Pull Request للمستودع الأصلي.
________________________________________
